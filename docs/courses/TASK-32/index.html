TASK-32 — The Two Laws
Module 1 · The Paper Trail

If it isn't written down somewhere checkable, it didn't happen

Imagine a company where deals are closed with a handshake in the hallway and nobody writes anything down. A month later, nobody agrees on what was promised. TASK-32 wrote down praxis's first law against exactly that — but for AI-assisted work: neither the AI nor the person steering it ever does anything without leaving a durable paper trail, or without checking against real evidence already in the project — a file, a git commit, a task on the board. The rule got a permanent home: a new document, docs/principles.md, plus a copy planted into every project that bootstraps onto the praxis lifecycle.

🗒️

Chat is not a record

A choice that lives only in a conversation turn did not happen. Decisions must land in artifacts that outlive the conversation and can be reviewed by a human later.

🔁

Artifacts in, artifacts out

Decisions are derived from what already exists on disk and on the board — and every decision produces new evidence in turn.

🙅

Don't re-ask the answered

If an existing document or principle already answers a question, the answer is taken from it — not re-asked as a matter of taste.

The law checks itself

Here is the pleasing part: the principle about programmatic evidence is itself guarded by a programmatic check. A new automated test refuses to pass unless the canonical document exists and the planted rulebook actually carries both laws — so the principles can never quietly fall out of what gets planted:

CODE 

assert.ok(existsSync("docs/principles.md"));
assert.match(TEMPLATE, /Artifact-grounded action:/);
assert.match(TEMPLATE, /One TASK, one PR:/);
PLAIN ENGLISH

The canonical principles document must exist on disk — not in anyone’s memory.

The rulebook planted into every new project must state the first law, word for word.

And it must state the second law too — a build that drops either one fails loudly.

🌍
Written once, inherited everywhere

The full statement lives in docs/principles.md — the canonical version, with rationale. The planted project rulebook carries a compact operational copy under "Rules that always hold." And a downstream project (Coda's constitution) now references the praxis document as its upstream source instead of re-deriving the argument — one law, one home, many inheritors.

Module 2 · One Task, One PR

The review boundary is the promise boundary

The second law is about granularity — how work is sliced. Every task system on earth distinguishes two levels, whatever it calls them: the TASK, a top-level deliverable someone committed to, and the SUBTASK, the internal breakdown of how that deliverable gets done. The law fixes the relationship between them: a TASK maps one-to-one to a pull request; a SUBTASK never gets its own. Subtasks land as ordinary commits on the parent task's single branch and merge together in that task's one PR.

The wording in docs/principles.md is deliberately tool-agnostic, because the disguises vary: Backlog.md spells subtasks as dotted ids (TASK-3.02), GitHub spells them as sub-issues or checklist items, Jira calls them sub-tasks, Spec Kit slices a spec into phases. Different costumes, same invariant.

⚖️
Why it matters

A pull request is where a human reviews and approves work. If review happens at the level of internal chunks, the actual promise — the deliverable — is never reviewed as a whole. And the law cuts both ways: a subtask that seems to deserve its own PR is really a deliverable in disguise (promote it on the board first — a paper-trail move, per the first law), while a task too big for one reviewable PR is mis-scoped and should be split as a task, not smeared across several PRs.

Overheard on a project

🛠️
Builder

I finished subtask 3.02 — the branch-prefix check. Opening a little PR for it now?

📜
The rulebook

No — 3.02 is a subtask. It rides the parent task’s branch as a commit. Task 3 gets exactly one PR, and everything inside it merges together.

🛠️
Builder

But this piece feels big enough to review on its own…

📜
The rulebook

Then it might really be a deliverable. Make that call where deliverables live — promote it to a full task on the board, then it earns a branch and a PR. Branch mechanics never restructure the plan of record.

🛠️
Builder

And if the parent task balloons and its one PR turns huge?

📜
The rulebook

That’s a scoping smell, not a license for multi-PR tasks. Split the task. The one-to-one mapping stays.

The law, as this very task obeyed it

TASK-32 itself followed the second law. Its whole life on the command line is three moves — and the middle one repeats as often as needed while the other two happen exactly once:

CODE 

git checkout -b task-32-101-principles
git commit -m "TASK-32: ..."   # once per slice
gh pr create                     # exactly once
PLAIN ENGLISH

One task, one branch — created when the task starts, named after it.

Every slice of internal progress becomes a commit on that same branch — as many as the work needs.

And exactly one pull request carries the whole deliverable to review.
