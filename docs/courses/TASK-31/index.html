TASK-31 — Setting the Table
Module 1 · The Empty Slot

A toolbox can't hang its own rules on your wall

The praxisflux plugins are like toolboxes: they sit closed until a job needs them. Nothing in a closed toolbox is always visible — yet the whole suite depends on standing rules the project must see every single session: which folders mean what, which gates apply, how work gets handed between tools. TASK-31 built the suite-level answer: one skill, pdlc:bootstrap, that stamps those standing rules onto any project — a brand-new empty folder or a codebase with years of history.

🌱

Day zero or year five

An empty folder gets a fresh rulebook; an existing project gets the rules added after its own — never over them.

🔖

One marked block

Everything planted lives between two visible bookmarks in the rulebook. Inside: refreshable boilerplate. Outside: yours, untouchable.

🤖

The boring part is a machine

A small deterministic script does the planting, byte-for-byte repeatable. The AI contributes judgment — questions, diffs, consent — never the typing.

The five verdicts, verbatim

The heart of the planting script is one decision: what state is the project's rulebook in, and what is the honest move? Five verdicts cover every case:

CODE 

if (!existsSync(claudePath)) { claudeMd = "created"; }
else if (!found) { claudeMd = "appended"; }
else if (found.block === expected) { claudeMd = "unchanged"; }
else if (!force) { claudeMd = "drifted"; }
else { claudeMd = "replaced"; }
PLAIN ENGLISH

No rulebook at all? Write a fresh one.

A rulebook exists but has no bookmarks? Add the block at the end — the owner’s text stays first and stays whole.

The block matches exactly what this version would plant? Touch nothing.

The block differs and nobody has consented? Report “drifted” and stop — differing text is never silently overwritten.

Consent given? Replace just the block, leaving everything around it alone.

🔖
Why bookmarks change everything

Because the planted rules live between visible markers, three hard problems become easy: an existing rulebook is appended to instead of clobbered, an upgrade can refresh the whole block in one honest move, and "your edits" versus "our boilerplate" has a physical boundary instead of a judgment call.

Why does the planted grounding live between two marker comments instead of being its own file content?

Module 2 · Guests, Not Squatters

Two invited companions and a doorbell

The lifecycle works best alongside two outside tools: Backlog.md (the task board) and GitHub Spec Kit (feature specs). Neither belongs to praxisflux — they are officially supported peers, and TASK-31 made the relationship explicit. Bootstrap never assumes them, never installs them behind your back, and never squats on a setup you already have. It rings the doorbell:

🔍
1 · Look before asking

The skill checks whether each companion's command-line tool is installed on your machine. That's the fork in the road.

💬
2 · Absent? Recommend, don't require

You're told the tool is officially supported and given the install command. Declining is fine — the project is planted without that companion's rules, and opting in later is one re-run away.

🚪
3 · Present? Ask to opt in

Having a tool installed doesn't mean this project should use it. One question per companion — and on an update, your previous answer is the remembered default.

🤝
4 · Opting in means bootstrap does the setup

It runs the companion's own initializer for you — and skips it entirely if the folder shows the companion is already set up. Your answers select which convention blocks get planted, and the stamp file remembers them.

The stamp that refuses to lie

Every bootstrapped project carries a small stamp file recording which plugin version planted it and which companions it opted into. One guard keeps the stamp honest:

CODE 

// A drifted, unconfirmed block means nothing was planted —
// don't advance the sentinel past it.
if (!check && !same && claudeMd !== "drifted") {
  writeFileSync(sentinelPath, JSON.stringify({ ...desired, plantedAt }));
}

…so the stamp never claims a state the folder isn’t actually in.

💡
Key Insight — set the table, don't cook

Bootstrap creates no wiki, no course, no lessons, and never invokes the plugins that do. It plants the rules, wires the transport, and hands off — telling you what's now possible (ground the codebase, spec the first feature) without doing any of it. In this suite, tools compose through files and gates, never by calling each other.
