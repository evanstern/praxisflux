The Paved Road — How praxisflux Scaffolds a New Plugin
Module 1

Five Hand-Edits and a Wall of Inspectors

praxisflux is a plugin marketplace: one repository holding eight plugins that share one chassis. This course is about a single question: what did it take to add plugin number nine?

Until this task, the honest answer was: five careful hand-edits in five different places — and if you missed one, a robot rejected your work. Let's meet the robots first.

The inspectors: drift gates

praxisflux runs on a rule: documentation and catalogs are never allowed to drift from what the code actually ships. The rule is enforced by drift gates — think of them as building inspectors who visit after every change. Each inspector checks one kind of agreement:

📋
check-docs

Does the README name every plugin the marketplace ships — a table row and an install line for each?

🗂️
gen-marketplace

Does the marketplace catalog match each plugin’s own manifest?

🔢
sync-version

Does every plugin carry the exact same version number as the marketplace? One release, one number, everywhere.

🧪
the test suite

Do all 140+ automated checks still pass with the new plugin present?

🏛️
Why so strict?

The repo’s founding principle is that a claim can never exceed its evidence. A README that lists a plugin that doesn’t exist — or misses one that does — is a false claim, so a gate blocks it.

What an inspector actually looks like

Here is the real heart of check-docs. It loops over every plugin registered in the marketplace and demands the README mention each one — twice:

scripts/check-docs.mjs 

for (const p of mp.plugins ?? []) {
  if (!readme.includes(`| **${p.name}** |`))
    problems.push(`README.md: plugin '${p.name}' has no row in the plugins table`);
  if (!readme.includes(`/plugin install ${p.name}@praxisflux`))
    problems.push(`README.md: plugin '${p.name}' has no '/plugin install ${p.name}@praxisflux' line`);
}
PLAIN ENGLISH

Go through every plugin the marketplace catalog lists...

If the README table has no row for this plugin...

...write down a complaint naming exactly what is missing.

If the README has no install command for this plugin either...

...write down a second complaint, just as specific.

Then move on to the next plugin.

Notice the inspector doesn’t fix anything — it only names problems. Fixing was the human’s job. That’s the pain this task removes.

The insight that became this task

The inspectors already know exactly what a finished plugin must look like. So why is a human assembling it from memory? Anything a gate can check, a tool can generate.

Module 2

The Scaffolder

TASK-36’s answer is one new tool: a scaffolder. One command replaces all five hand-edits:

node scripts/new-plugin.mjs my-plugin Stamp everything the inspectors will demand — in one shot

Think of a housing development where every lot used to be built from a hand-drawn sketch — and the city inspector rejected half of them. The scaffolder is the pre-fabricated frame: it arrives already meeting code, and the builder only does the parts that need judgment — what the house is for.

🛣️
The paved road

Engineers call this a paved road: the easy way and the correct way become the same way. You can still walk off-road — the scaffolder just makes the right path the effortless one.

What one command stamps

Everything below appears from a single run — each piece aimed at a specific inspector from Module 1:

my-plugin/ the new plugin, ready for the inspectors
.claude-plugin/plugin.json the manifest — version copied from the marketplace, so sync-version passes
skills/my-plugin/SKILL.md a skill skeleton with the sections and header fields the gates expect
lib → ../lib a symlink to the shared chassis
.claude-plugin/marketplace.json catalog entry added automatically — gen-marketplace passes
README.md table row + install line inserted — check-docs passes

Where judgment is required — the plugin’s description, what its skill actually does — the scaffolder writes a loud TODO instead of guessing. Machines stamp the checkable; humans write the meaningful.

The guards, in real code

The very first lines of the scaffolder are refusals. Nothing is written until both guards pass — so running it twice can never destroy work:

scripts/new-plugin.mjs 

if (!/^[a-z][a-z0-9]*(-[a-z0-9]+)*$/.test(name))
  throw new Error(`plugin name must be kebab-case (got ${JSON.stringify(name)})`);
const dir = join(repo, name);
if (existsSync(dir)) throw new Error(`${name}/ already exists — refusing to clobber`);

const mp = JSON.parse(readFileSync(join(repo, ".claude-plugin", "marketplace.json"), "utf8"));
PLAIN ENGLISH

Is the name in kebab-case? If not...

...stop immediately and say exactly what was wrong with it.

Work out where the new plugin folder would go.

If that folder already exists, stop — never overwrite someone’s work.

(Only past both guards do we proceed...)

...and the first real act is reading the marketplace, so the new plugin can copy its version number.

🔒
Fail-safe by ordering

The guards run before any file is written. Order is the whole safety mechanism: a tool that checks after writing has already done the damage.

The optional extra: --with-gate

Some plugins enforce their own rules with a Stop hook. Adding the flag --with-gate stamps that whole enforcement kit too:

🚦

gates/my-plugin.mjs

The rule-checker itself. Stamped as a stub that checks nothing yet — harmless until you wire in real rules.

🔌

scripts/stop.mjs + gate.sh

The wiring that connects the rule-checker to the Stop hook, borrowed line-for-line from the plugins that already work.

📜

hooks/hooks.json

The registration card telling Claude Code the hook exists and when to run it.

The stub gate deliberately applies to zero projects until someone edits it. A brand-new plugin should never block anyone’s work by accident.

Module 3

Proving It Works

A tool that claims to satisfy the inspectors is worth nothing in this repo — claims can’t exceed evidence. So the scaffolder ships with proof, gathered two ways.

🧪
Way 1: automated tests in a fixture repo

A disposable miniature marketplace, rebuilt from scratch for every test run

🏟️
Way 2: a live dress rehearsal in the real repo

Scaffold a scratch plugin for real, run every inspector, then remove it

Way 1: the crash-test town

Car makers don’t test airbags on city streets — they build a crash-test facility. The tests here do the same: each one builds a fixture marketplace — a fake README, catalog, and one existing plugin — scaffolds into that, and then unleashes the real inspectors on the result:

test/new-plugin.test.mjs 

scaffoldPlugin(root, "newbie");

// check-docs: README row + install line were inserted.
assert.deepEqual(checkDocs(root), []);
PLAIN ENGLISH

Scaffold a plugin called “newbie” into the crash-test town.

Then...

...invite the README inspector from Module 1 — the real one, not a copy...

...and insist its list of complaints is exactly empty.

Other tests in the same file prove the rerun guard (scaffold twice → error, files untouched), the --with-gate kit, and the name check. The test suite runs before every commit, so this proof re-runs forever.

Way 2: the dress rehearsal

Tests in a fixture are necessary but not sufficient — so the scaffolder was also run once, for real, in the actual repo:

1
Scaffold a scratch plugin

node scripts/new-plugin.mjs scratch-demo --with-gate — the full kit, stamped into the live repo

2
Summon every inspector

check-docs, gen-marketplace, sync-version, and the full 142-test suite — all four passed with zero edits to the stamped files

3
Strike the set

Delete the scratch plugin and restore the catalog and README — the rehearsal leaves no trace in the shipped repo

🎯
The universal lesson

Correct by construction beats correct by inspection. When a checklist matters, don’t just check it — build the tool that makes passing it the default. The inspectors still run, but now they’re confirming, not catching.

Closing the loop

One last edit made the change stick: the authoring guide’s new-plugin checklist — the very list that used to mean five hand-edits — now opens by naming the scaffolder as the paved path, and marks which steps it stamps versus which still need a human. The next author reads the better way first.

And the story has a planned ending: the ninth plugin, whenever it comes, is meant to be this scaffolder’s first real customer.
