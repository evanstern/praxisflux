TASK-20 — Teaching a Skill to Run Itself
Module 1 · The Question

Could nobody be there?

Every task in this project ends the same way: a person watches an AI assistant do the work in a chat window, reads along, nudges it, and decides when it is finished. TASK-20 asked a simple, slightly unsettling question: what if nobody was watching?

Could a skill run start-to-finish with no human in the chat — and could we still trust the result?

🤖
"Headless," in plain terms

Running a program headless means running it with no screen and no person attached — fire it off, walk away, collect the result. The "head" that is missing is you.

Why bother?

Because if a skill can run unattended, a whole workflow can be wired together like a factory line — with humans only at the decision points that genuinely need them. That is the project's larger plan, and it needs three kinds of stations:

⚙️

Machine steps

Boring, exact work — checks and bookkeeping — done by small programs that behave the same way every time.

🤖

AI steps

The genuinely creative middle — writing, deciding, fixing — done by an AI session running a skill. This is the step TASK-20 had to prove.

🙋

Human steps

Real judgment calls — approvals and merges — kept for people, on purpose.

The autopilot rule

Think of an autopilot flight. The airline does not decide the landing went well because the autopilot says "landing complete." It checks the instruments, the flight recorder, the aircraft itself. The report is not the evidence — the aircraft is.

This project already lives by that rule. Every claim of progress must be backed by artifacts, and small programs called gates inspect those files and reject anything unproven. TASK-20's bet: that same rule is exactly what makes an unattended AI run trustworthy.

The inspection, in real code

Here is the actual post-flight inspection from TASK-20 — two commands, run after the AI session ends. They are the entire basis for trusting the run:

CODE 

node <praxisflux>/spec-bridge/gates/cli.mjs plan .    # must print nothing
node <praxisflux>/spec-bridge/gates/cli.mjs check .   # must exit 0, no lag warnings
PLAIN ENGLISH

Ask the checker: is any work still left undone here? Silence means the job is fully reconciled.

Then ask the gate: does every claimed status match the actual files? A clean pass is the only verdict that counts.

💡
Key Insight

Trust the evidence, never the narrator. Engineers call this verification by artifact — and it is the single idea that makes unattended automation safe enough to use.

An unattended AI run finishes and reports "all done, everything went great." What should decide whether the work is actually accepted?

Module 2 · The Test Flights

Three runs, nobody watching

A claim like "AI can run unattended" is worthless without a test. So TASK-20 built a small throwaway project — a practice dummy with a task board and a work plan — and launched the real skill at it three times, using one command.

The launch command

This is the actual command, verbatim. One line of it does almost all the talking:

CODE 

claude -p "/spec-bridge:sync" \
  --allowedTools "Bash(node:*),Bash(backlog:*),Bash(git:*),Read,Glob,Grep" \
  --output-format json > run.json
PLAIN ENGLISH

Start an AI session with no chat window, and hand it one instruction: run the board-syncing skill.

Grant it a specific toolbox — it may run these named programs and read files, nothing we did not list.

When it finishes, save a machine-readable receipt (cost, timing, result) into a file for inspection.

The flag -p is the whole trick: it tells the CLI to run one instruction to completion and exit — no conversation, no waiting for a human.

The three flights

1
The catch-up

The board was behind the real work. Unattended, the skill moved the task forward and mirrored the plan onto it. 8 turns · 49 seconds · $0.71

2
The finish line

The work plan was regenerated and everything checked off. The skill replaced the stale plan items and moved the task to Done — with a written summary derived from the files. 6 turns · 43 seconds · $0.67

3
The honest retreat

New unfinished work appeared after Done. The skill moved the task backwards — from Done to In Progress — because the files no longer proved Done. 10 turns · 45 seconds · $0.75

💡
Key Insight

Flight three is the one that matters. Any system can march forward; a trustworthy one retreats when the evidence stops supporting its status. No human told it to — the rule is in the files.

Module 3 · What We Learned

The surprises are the deliverable

The spike proved the happy path — but its real value is three findings that were not obvious going in. Each one changes how you should think about automating AI work, in this project or your own.

Finding 1: exit code 0 is not a gold star

Every run reported success the same way — even a probe that was set up to be under-equipped for the job. Here is a real receipt, straight from a run's saved output:

CODE 

{
  "is_error": false,
  "subtype": "success",
  "num_turns": 10,
  "total_cost_usd": 0.754
}
PLAIN ENGLISH

The receipt for one run, written as structured data…

"No error happened" — meaning the session ran without crashing…

"success" — but careful: this only means the session finished, not that the work is right…

The AI took ten back-and-forth steps with its tools…

…and the whole run cost about 75 cents.

End of the receipt.

⚠️
Common mistake

Automation that treats "the program exited cleanly" as "the task succeeded." A finished session and a correct result are different facts — only the gate inspecting the files can confirm the second one.

Finding 2: the toolbox list adds, it does not fence

The biggest surprise. One probe deliberately left the important tools OFF the allowed list — and the run used them anyway, freely. Why? The machine it ran on already had its own standing permissions, and the launch command's list only adds to those. It is not a fence; it is a gift registry.

The lesson for any real deployment: true limits must come from the environment — a sealed container with a clean configuration — not from the launch command's options.

Finding 3: the robot borrowed the house keys

The test project was empty, yet the skill was found and ran perfectly. How? The machine had the plugin installed, and the unattended session quietly inherited it. Convenient on a laptop — but a trap for automation: a fresh machine in the cloud has none of that. Whatever the run needs must be packed in explicitly.

The pre-flight checklist

The findings boil down to a checklist any skill must pass before it can run unattended. This is now a permanent document in the project:

1 · ends with proof The skill finishes by verifying its own work against files — never by asking a person a question
2 · decisions from files Every choice it needs is readable from the project — anything needing human judgment is split out as a human step
3 · fails loudly When it cannot finish, the leftover evidence must make the gate fail — a polite apology in the transcript helps nobody
4 · declares its needs Tools and plugins are named so a container can pack them — never assumed from someone's laptop
5 · machine core, AI shell Everything mechanical is a plain program; the AI handles only the genuinely creative part — cheaper, steadier runs
6 · scoped writes It is declared up front which files the skill may change, so the run can be audited afterwards
💡
Key Insight

Nothing on this checklist is about making the AI smarter. Unattended automation is earned with structure — proof at the end, limits from the environment, needs declared up front. The intelligence was never the bottleneck.

With the question answered and the checklist written, the path is open for the next step: wiring these unattended runs into a real production line, with the gates as checkpoints and a person holding the final approval.
