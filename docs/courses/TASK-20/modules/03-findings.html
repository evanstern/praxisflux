Module 3 · What We Learned

The surprises are the deliverable

The spike proved the happy path — but its real value is three findings that were not obvious going in. Each one changes how you should think about automating AI work, in this project or your own.

Finding 1: exit code 0 is not a gold star

Every run reported success the same way — even a probe that was set up to be under-equipped for the job. Here is a real receipt, straight from a run's saved output:

CODE 

{
  "is_error": false,
  "subtype": "success",
  "num_turns": 10,
  "total_cost_usd": 0.754
}
PLAIN ENGLISH

The receipt for one run, written as structured data…

"No error happened" — meaning the session ran without crashing…

"success" — but careful: this only means the session finished, not that the work is right…

The AI took ten back-and-forth steps with its tools…

…and the whole run cost about 75 cents.

End of the receipt.

⚠️
Common mistake

Automation that treats "the program exited cleanly" as "the task succeeded." A finished session and a correct result are different facts — only the gate inspecting the files can confirm the second one.

Finding 2: the toolbox list adds, it does not fence

The biggest surprise. One probe deliberately left the important tools OFF the allowed list — and the run used them anyway, freely. Why? The machine it ran on already had its own standing permissions, and the launch command's list only adds to those. It is not a fence; it is a gift registry.

The lesson for any real deployment: true limits must come from the environment — a sealed container with a clean configuration — not from the launch command's options.

Finding 3: the robot borrowed the house keys

The test project was empty, yet the skill was found and ran perfectly. How? The machine had the plugin installed, and the unattended session quietly inherited it. Convenient on a laptop — but a trap for automation: a fresh machine in the cloud has none of that. Whatever the run needs must be packed in explicitly.

The pre-flight checklist

The findings boil down to a checklist any skill must pass before it can run unattended. This is now a permanent document in the project:

1 · ends with proof The skill finishes by verifying its own work against files — never by asking a person a question
2 · decisions from files Every choice it needs is readable from the project — anything needing human judgment is split out as a human step
3 · fails loudly When it cannot finish, the leftover evidence must make the gate fail — a polite apology in the transcript helps nobody
4 · declares its needs Tools and plugins are named so a container can pack them — never assumed from someone's laptop
5 · machine core, AI shell Everything mechanical is a plain program; the AI handles only the genuinely creative part — cheaper, steadier runs
6 · scoped writes It is declared up front which files the skill may change, so the run can be audited afterwards
💡
Key Insight

Nothing on this checklist is about making the AI smarter. Unattended automation is earned with structure — proof at the end, limits from the environment, needs declared up front. The intelligence was never the bottleneck.

With the question answered and the checklist written, the path is open for the next step: wiring these unattended runs into a real production line, with the gates as checkpoints and a person holding the final approval.
