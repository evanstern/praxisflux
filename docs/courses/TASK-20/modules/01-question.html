Module 1 · The Question

Could nobody be there?

Every task in this project ends the same way: a person watches an AI assistant do the work in a chat window, reads along, nudges it, and decides when it is finished. TASK-20 asked a simple, slightly unsettling question: what if nobody was watching?

Could a skill run start-to-finish with no human in the chat — and could we still trust the result?

🤖
"Headless," in plain terms

Running a program headless means running it with no screen and no person attached — fire it off, walk away, collect the result. The "head" that is missing is you.

Why bother?

Because if a skill can run unattended, a whole workflow can be wired together like a factory line — with humans only at the decision points that genuinely need them. That is the project's larger plan, and it needs three kinds of stations:

⚙️

Machine steps

Boring, exact work — checks and bookkeeping — done by small programs that behave the same way every time.

🤖

AI steps

The genuinely creative middle — writing, deciding, fixing — done by an AI session running a skill. This is the step TASK-20 had to prove.

🙋

Human steps

Real judgment calls — approvals and merges — kept for people, on purpose.

The autopilot rule

Think of an autopilot flight. The airline does not decide the landing went well because the autopilot says "landing complete." It checks the instruments, the flight recorder, the aircraft itself. The report is not the evidence — the aircraft is.

This project already lives by that rule. Every claim of progress must be backed by artifacts, and small programs called gates inspect those files and reject anything unproven. TASK-20's bet: that same rule is exactly what makes an unattended AI run trustworthy.

The inspection, in real code

Here is the actual post-flight inspection from TASK-20 — two commands, run after the AI session ends. They are the entire basis for trusting the run:

CODE 

node <praxisflux>/spec-bridge/gates/cli.mjs plan .    # must print nothing
node <praxisflux>/spec-bridge/gates/cli.mjs check .   # must exit 0, no lag warnings
PLAIN ENGLISH

Ask the checker: is any work still left undone here? Silence means the job is fully reconciled.

Then ask the gate: does every claimed status match the actual files? A clean pass is the only verdict that counts.

💡
Key Insight

Trust the evidence, never the narrator. Engineers call this verification by artifact — and it is the single idea that makes unattended automation safe enough to use.

An unattended AI run finishes and reports "all done, everything went great." What should decide whether the work is actually accepted?

In the factory-line picture of this workflow, where do humans deliberately remain?
