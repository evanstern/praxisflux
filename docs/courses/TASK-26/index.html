TASK-26 — Nobody Else's Kitchen
Module 1 · The Shared Kitchen

The incident that wrote this task

During the pipeline's very first week, an AI agent was two minutes into implementing a feature when a human — juggling three things at once — switched the project's active branch out from under it. The agent's half-written work was suddenly sitting on the wrong branch.

It got caught in seconds and recovered. But "we noticed fast" is not a safety model. The root cause: the pipeline gave each run a fresh branch, but every run — and every human — shared one working copy of the project. One kitchen, many cooks, no doors.

🔪
The trap: isolation by politeness

"Nobody will touch the folder during a run" is a rule enforced by hope. The moment automation shares a workspace with humans — or with other automation — collisions stop being if and become when.

The fix: every run gets its own kitchen

Git has a built-in answer: a worktree — a second (or tenth) working folder wired to the same project history. Each pipeline run now cooks in its own:

The whole fix is three lines

CODE 

const wt = join(mkdtempSync(join(tmpdir(), "n8n-pilot-wt-")), "run");
execFileSync("git", ["-C", target, "worktree", "add", "-b", branch, wt, "main"]);
runs.set(runId, { dir: wt, target, branch, agentRounds: 0 });
PLAIN ENGLISH

Pick a fresh throwaway folder for this run alone.

Ask git to set up a private working copy there, on a brand-new branch, without disturbing the original.

Remember: this run lives in THAT folder from now on — every later step works there.

The original design gave each run its own branch — and the incident happened anyway. Why wasn't a branch enough?

Module 2 · Landing Rights

Refuse loudly, keep the work

Isolation solves the working half. The landing half — merging an approved run back into the project — still touches shared ground: the target's own main branch. What if, at that exact moment, the target is sitting on some other branch, or has uncommitted edits?

The old code would have switched branches under whoever was there — the incident, but in reverse. The new landing rule is the opposite of force:

✅
Target on a clean main → land

Merge the run's branch (keeping its history visible), remove the worktree, delete the branch. Leave no litter.

🛑
Anything else → refuse, loudly, safely

Fail with a message naming exactly what to fix — and keep the run's worktree intact, so approved work is never lost to a refusal.

The refusal is safe to be strict about for one structural reason: landing only ever happens at the human-approval step. If the target is in a weird state, a person is right there to fix it and re-approve.

CODE 

if (targetBranch !== "main")
  throw new Error(`target is on '${targetBranch}', not main — check out a clean main in ${run.target}, then re-approve. Run's work is safe on ${run.branch} (worktree kept: ${run.dir})`);
PLAIN ENGLISH

If the project is not sitting on its main line when the merge is requested…

…refuse — and make the refusal a complete instruction: what is wrong, where to fix it, that the approved work is safe, and exactly where it is waiting.

Proven by replaying the crime

The task ships a self-contained repro script that spins up its own runner and re-enacts the original incident on a throwaway project. Its transcript, verbatim:

CODE 

ok   mid-run branch switch + dirty target: run tree unaffected
ok   two concurrent checkouts of the same target coexist
ok   finish refuses an unstable target and keeps the worktree
ok   finish merged --no-ff into target main, removed worktree + branch
ok   concurrent run 2 unaffected by run 1's landing
isolation: all checks passed
PLAIN ENGLISH

The original incident, replayed on purpose — and this time it is a non-event.

Two runs of the same project at once: both healthy, neither aware of the other.

A messy target at landing time gets a refusal, not a forced merge — and the work survives.

A clean target gets the real landing, with full cleanup after.

And landing one run leaves its neighbor untouched.

Every safety claim in this course is a passing check, not a promise.

Key Insight

When a safety fix comes from a real incident, encode the incident as a test. "We fixed it" fades; a check that replays the exact failure and passes is the fix, permanently on guard.
