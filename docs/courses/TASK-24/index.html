TASK-24 — The Silent Almost
Module 1 · The Silent Almost

Every box checked. Nothing happens.

This project's task board has a watchdog — a gate that compares every task's claimed status against what its files actually prove. It knew three situations:

⛔

Overclaiming

The board says Done but the files disagree. BLOCKED — loudly, immediately.

🐢

Lagging

The files are ahead of the board. A gentle warning: run the sync.

😶

Honest

Board and files agree. Silence — no news is good news. Usually.

The trap inside "honest"

The project has an optional strict mode: finishing a task requires not just every checkbox ticked, but a saved analysis report with no unresolved CRITICAL findings. Now imagine: every box checked, report missing. The task honestly says In Progress. The gate sees "honest" — and says nothing.

🕳️
The silent almost

Done is unreachable, and no part of the system says so. You finished everything on the list, and the finish line quietly moved — discoverable only if you happen to ask the right diagnostic command.

Where the finish line actually is

The rule that creates the trap is real code — the project's single source of truth for "how done is this spec":

CODE 

status = has("plan.md") && tasksTotal > 0 && tasksDone === tasksTotal && analysisOk
  ? STATUS.DONE_ELIGIBLE
  : STATUS.IN_PROGRESS;
PLAIN ENGLISH

A task can only finish when the plan exists, the work list is non-empty, every item is checked — AND the analysis requirement is satisfied.

All four true: eligible to be marked Done.

Any one false: still In Progress — including when the ONLY missing piece is that analysis report.

Every checkbox is done, the report is missing, and the board says In Progress. Why doesn't the gate complain?

Module 2 · Warn, Never Block

A smoke detector, not a locked door

The fix is one carefully-aimed rule. When a task is honest AND every box is checked AND the only thing between it and the finish line is the analysis requirement — the gate now speaks up. On the warning channel. Never the blocking one.

Why never block? Because blocking is for dishonesty, and this task is telling the truth. What it needs is a smoke detector — something that says "you're one report away, here's exactly what to do" — not a locked door punishing it for a state it honestly declared.

The new rule, verbatim

CODE 

v === "ok" &&
derived.analysis?.required &&
derived.status !== STATUS.DONE_ELIGIBLE &&
derived.tasksTotal > 0 &&
derived.tasksDone === derived.tasksTotal &&
existsSync(join(root, task.specDir, "plan.md"))
PLAIN ENGLISH

Only for tasks whose status is honest — liars are already handled by the blocker…

…and only when strict mode is actually demanding an analysis report…

…and the task has not reached the finish line…

…despite the work list being real…

…and every single item on it being checked…

…and the plan existing too — so the report is provably the ONLY missing piece.

Six conditions, and every one earns its place: drop any of them and the warning would fire in situations where the report isn't the only blocker — turning a precise nudge into noise people learn to ignore.

Key Insight

Good automation separates its voices: one channel that stops you (for things that must not happen) and one that informs you (for things you'd want to know). Collapse them and people either get blocked unfairly or learn to ignore everything.
