TASK-35 — The Transplant & the Third Shape
Module 1 · The Transplant

A battle-tested skill moves house — and nothing has to be rewritten

Outside this repo lived a skill called team-review: it runs a codebase review the way an engineering lead runs one, sending several specialist AI teammates into the code in parallel and merging their findings into one evidence-backed report. It had been hardened through three graded practice rounds — scoring a perfect 10/10 unsupervised on the final one. TASK-35 brought it home: turned the standalone skill into the marketplace's eighth plugin, team-review/.

The remarkable part is what didn't change. The skill had been built from day one to speak this repo's conventions — same gate contract, same directory rules, same storage transport — so adopting it was a move, not a rewrite. The only surgery was swapping its small vendored copy of the stop-hook machinery for the shared original it had been imitating.

🧳

Built to move

The standalone skill vendored a ~30-line imitation of the shared gate-runner, matching its contract exactly — so the transplant could delete the copy and import the real thing.

🚦

Gates never write

The house rule held on arrival: the plugin's gates/ folder only checks things. A search for any file-writing call across it comes back empty — provably.

📦

Proven installed

The finished plugin was packaged exactly as the marketplace installs it, then driven end-to-end against a sample repo — which came out byte-for-byte untouched.

The one real change, in three lines

After the move, the plugin's stop-hook entry — the tripwire that refuses to let a session end while a review is unfinished — collapsed to almost nothing, because the shared chassis now does the heavy lifting:

CODE 

import { runStopHook } from "../lib/gate-runner.mjs";
import { reviewGate } from "../gates/review.mjs";
runStopHook({ gates: [reviewGate] });
PLAIN ENGLISH

Borrow the shared machinery every plugin in this marketplace uses to guard the end of a session.

Bring team-review’s one domain rule: an open review must be finished or formally closed, never abandoned silently.

Wire the two together — that single line is now the entire stop hook.

🧾
The spec came from the patient

Before moving in, the skill was pointed at this very repo and asked for a review. Its report — vendored into the repo at docs/handoffs/team-review-iteration-3-review.md — became the task's design input: it named exactly which two potholes a newcomer plugin would hit, and the transplant plan was written around fixing them first. Module 3 tells that story.

Module 2 · The Third Shape

A guest in someone else's house: review the repo, touch nothing

Every plugin in this marketplace "lives" somewhere. Until now there were two shapes: plugins with a favored home (a folder marked as theirs, like a classroom), and drop-anywhere plugins (any folder can become theirs by planting a marker file). team-review fits neither. It works on a repo the caller points at — someone else's project — and the whole engagement is read-only by doctrine. It may not leave so much as a settings file behind in the repo it reviews.

So where does its bookkeeping go? At the invoking project's root — the place you're working from, never the place you're pointing at. Run records ride the gitignored .handoff/ transport there; the report lands outside the target too. TASK-35 named this the third placement model and wrote it into the authoring guide, so the next plugin like this doesn't have to rediscover the rule.

The rule that keeps the guest honest

The authoring guide's new paragraph carries one load-bearing warning. The shared helper that sets up the .handoff/ storage also edits a project's .gitignore — a write. Pointed at the invoking root, that's correct plumbing. Pointed at the review target, it would itself be the forbidden write. The guide now states the rooting rule inline:

CODE 

// skill-patterns §6 — third placement model
ensureGitignore(invokingRoot)   // ✓ correct transport
ensureGitignore(reviewTarget)   // ✗ the forbidden write
PLAIN ENGLISH

The guide names the shape: operate on a caller-supplied repo, keep state at your own root.

Setting up storage at your own root is legitimate housekeeping.

Doing the same to the repo you’re reviewing would modify it — exactly what a review must never do. Nothing gets installed into the target: no hook, no rulebook, nothing.

🕵️
Walking away is also caught

Open a review and try to end the session without finishing it? The plugin's stop hook blocks the exit, listing what's missing and the two honest ways out: finish (produce a passing report) or abandon (close it formally, with the reason kept on record). A review that's out of scope for your current folder never triggers it — no runs in scope, no opinion.

Module 3 · Paving the Road

The new arrival reviewed its own destination — and the potholes got fixed first

Here's the loop that makes TASK-35 unusual: before the transplant, the team-review skill was aimed at this repo — the marketplace it was about to join — with the question "how ready is this chassis for a plugin like me?" Its 10/10-gated report found the two potholes a newcomer would hit first, and the task was deliberately sequenced to pave the road before driving on it: fix both, then move in as the first beneficiary.

The generator, before and after

The pothole fix itself is small enough to read. The catalog generator used to loop only over plugins already registered; now it also sweeps the repo's top-level folders and adopts any that carry a plugin manifest:

CODE 

for (const e of readdirSync(repo)) {
  if (registered.has(e.name)) continue;
  const pj = manifest(e.name);
  if (pj) plugins.push({ name: pj.name, ... });
}
PLAIN ENGLISH

Walk every folder at the top of the repo, not just the ones the catalog already knows.

Skip anything that's already registered — existing entries keep their hand-tuned settings.

Peek inside for a plugin manifest — the file that says "I am a plugin."

Found one that's new? Write it into the catalog with sensible defaults.

That's the whole fix: the checklist's "just run the generator" is now literally true.
