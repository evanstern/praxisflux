TASK-21 — The Librarian Gets a Sorting Machine
Module 1 · The Bookkeeping Tax

The wiki that refuses to lie

This project keeps a living encyclopedia of itself — a wiki where every note carries two receipts: which source files it describes, and a pin naming the exact code snapshot it was last checked against.

A gate enforces the deal: if any source file changed after a note's pin, the note is declared stale — and nothing ships until someone re-verifies it.

📚
Why so strict?

Documentation that might be current is worse than none — you cannot tell which parts to trust. The pin turns "probably still true" into "checked against exactly this version."

The tax

Strictness has a price. On one busy day in this project, the freshness check fired on eight notes at once — because a routine release had touched a version number inside files those notes listed as sources. The change was one digit. The follow-up was not:

1
Read eight change reports

Open the history for each note and confirm what actually changed in its sources

2
Discover most were trivial

A version number ticked up — the notes' explanations were untouched by it

3
Re-pin all eight by hand

Edit each note's pin line to the new snapshot — carefully, because a wrong pin is a false receipt

That is skilled attention spent on filing. And this project had just learned a name for the cure, from its task-board tooling: computed, not reasoned — when the follow-up to a change can be calculated, a program should calculate it, and thinking should be saved for the parts that need it.

What "stale" looks like, in the real machinery

This is the actual heart of the freshness check — the question it asks the project's history for every note:

CODE 

const log = git(repoRoot, ["log", "--oneline", `${pin}..HEAD`, "--", ...sources]);
if (!log) continue; // fresh
PLAIN ENGLISH

Ask the project history: list every change to this note’s source files made after its pin.

Silence means nothing changed — the note is still fresh, move on to the next one.

TASK-21's question: when that list is not silent, could a program tell the trivial cases from the ones needing real judgment — and do the filing itself?

Module 2 · The Sorting Machine

plan: two piles, one honest rule

TASK-21 built a sorting machine. Point the new plan command at the wiki and it examines every stale note's changes, then sorts them into exactly two piles:

🗂️

RE-PIN-ONLY

The change provably cannot have touched what the note says. The machine prints a ready-to-run filing command — no thinking required.

🔍

NEEDS-REVIEW

Everything else. The machine prints a work order — what changed, how much, why it needs eyes — and a person (or AI) does the judging.

The one honest rule

What earns a spot in the easy pile? The machine allows exactly one provably-safe case, and here it is — the real classifier, verbatim:

CODE 

if (!changedLines.every((l) => STAMP_LINE.test(l)))
  return { cls: "REVIEW", reason: "sources changed beyond version stamps — re-verify the prose against the diff" };
if (SEMVER.test(String(noteBody ?? "")))
  return { cls: "REVIEW", reason: "stamp-only diff, but the note quotes version literals — update them, then re-pin" };
return { cls: "REPIN", reason: "version stamps only, and the note quotes no versions" };
PLAIN ENGLISH

If even one changed line is anything besides a routine version-number stamp…

…refuse to file it — send it to review with the reason spelled out.

Even if every change IS a version stamp: does the note itself mention any version number?

Then it might be quoting the very number that just changed — review, with instructions.

Only when both checks pass is the change provably trivial: file it automatically.

💡
Key Insight

The machine's power comes from what it refuses to do. When it cannot prove a change is trivial — including when it simply could not read the change — it defaults to the review pile. A filing system you can trust is one that never guesses.

The backticks near-miss

One bug was caught during the build, and it is worth retelling. Notes in this wiki quote version numbers inside backticks. The classifier's first draft stripped code formatting before scanning — which would have hidden exactly those version mentions and filed notes that needed review. The fix: scan the raw text, so any doubt pushes toward the review pile, never away from it.
