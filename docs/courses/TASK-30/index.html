TASK-30 — The System Builds Itself
Module 1 · Rules Before Robots

Letting the machine touch its own machinery

Self-hosting — a system developing itself — sounds reckless. TASK-30's bet was that it becomes safe when three rules are structural, not aspirational. All three were built and tested on other projects before the machine was allowed anywhere near its own:

🏝️

Work in isolation

The AI gets a private, disposable copy of the repository. Its mistakes are local until proven otherwise.

⚖️

Same law for everyone

The repo's hooks, gates, and house manual govern whoever works there — the machine gets no exemptions from rules it enforces on humans.

📮

Propose, never merge

Human approval creates a proposal (a pull request) — review and CI keep the final word, exactly as for any contributor.

📮
The subtle one: approval ≠ merge

On other projects the pipeline's approval merged directly. For its own repository, approval was deliberately demoted to "open the proposal." Self-hosting doesn't bypass the front door — it knocks like everyone else.

Module 2 · What the Bumps Taught

Six findings from one bumpy, successful run

The run worked — the feature landed as a reviewed pull request. But the retrospective is the real product. Six findings, each one general enough to matter beyond this project:

⏱️
1 · Time limits sized for small jobs kill big ones

The AI was terminated at ten minutes — twice. Real implementation work needs budgets sized per request, not one cap for everything.

💾
2 · Cheap deaths are a design property

Both kills lost almost nothing, because the instructions demanded committing every working slice immediately. Survivability was written into the prompt, not hoped for.

🔍
3 · The machine audited its own house

Working in isolation, the AI discovered the repo's pre-commit checks leaked repository identity into test sandboxes — a real bug nobody knew — and fixed the root cause unprompted.

😶
4 · Gate-passed is not work-complete

After the first kill, a half-finished-but-honest state PASSED the gate and parked for approval. The human window caught it — but detection shouldn't depend on that.

🚦
5 · The runner is single-threaded

Discovered when a simple status check hung: the runner can't answer anything while an AI round runs. Pilot-acceptable, production-absurd — now it's written down.

🛠️
6 · Humans can repair a parked run

Driving the runner directly — "continue where you were killed" — while the workflow stayed parked turned out to be a legitimate, graceful repair path. The pause is a feature twice over.

The continuation that saved the run

Finding 6 in action — the exact repair, sent straight to the runner while the workflow stayed parked:

CODE 

curl -s -X POST localhost:8787/agent -d '{"runId":"run-mrh660hx-adnns",
  "prompt":"CONTINUATION: you are resuming your own interrupted work…
  Commit in small slices NOW and often…
  Budget your time: committing working progress beats polishing."}'
PLAIN ENGLISH

Speak to the execution service directly — same run, same private workspace.

Tell the AI the truth: it died mid-work, and here is what survived.

Make survivability an instruction: every finished slice becomes permanent immediately.

And set the priority a timeout demands: done beats polished.

💡
Key Insight

A retrospective's findings are only real if they change something: three of these six are already fixes or conventions, and the other three are named follow-up work. A lesson without a landing place is just an anecdote.

A half-finished state passed the gate. What does that reveal about gates in general?

Humans repaired the run by driving the runner directly while the workflow stayed parked. Why did that work without confusing the workflow?
