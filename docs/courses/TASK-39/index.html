TASK-39 — Proving the Install Path
Module 1

The Untested Bet

praxisflux is a plugin marketplace: eight plugins sharing one toolbox of common code. An outside architecture review of the project ended with one worry it called the biggest structural bet, unverified: nobody had ever proven that installing a plugin actually works.

TASK-39 is the story of turning that bet into a checked fact. To see why it was a bet at all, you need to meet one strange character: the symlink.

The recipe card and the family cookbook

Imagine a box of recipe cards. To save space, every card that needs the house tomato sauce just says “sauce: see the family cookbook, page 12.” At home, perfect — the cookbook is always on the same shelf. But mail a card to a friend and the reference is worthless… unless whoever copies the card also copies page 12 onto it.

praxisflux works exactly like that box. The shared code lives once, in a folder called lib/, and every plugin contains a symlink named lib that points at ../lib — “one shelf up.”

praxisflux/ the whole marketplace repository
lib/ the family cookbook — shared code, kept once
team-review/ one of the eight plugins
lib → ../lib not a folder — a signpost saying “see the cookbook, one shelf up”
scripts/ this plugin’s own code, which follows that signpost

When you install a plugin, the marketplace copies just that plugin’s folder onto your machine — and it is supposed to dereference the signpost: copy page 12 onto the card. Supposed to. No test had ever checked.

What’s riding on it: the Stop hook

Why does it matter? Because the shared code being reached through that signpost includes the plugins’ conscience. Four praxisflux plugins ship a Stop hook: every time the AI assistant tries to end its turn, the hook inspects the project and refuses to let it finish if the work claims more than the files prove.

Here is the chain that fires on every single stop. Step through it:

An exit code is the whole conversation: 0 means “let the assistant finish,” 2 means “block, and show it my complaint.”

Borrow a tool for listing folder contents from Node itself — always available, no bet here.

Borrow a tool for gluing file paths together — also built in, also safe.

Now the bet: load the shared gate-runner from “../lib” — one folder up, then into lib. That path only works if the signpost was turned into a real folder during install.

Same bet again: the project-root finder also lives behind the signpost. Two of four imports depend on the copy step nobody had tested.

In the repository, ../lib resolves through the symlink. In an installed copy, it resolves only if the installer really did paste page 12 onto the card.

The scary part: failure would be silent

Suppose the install step didn’t dereference. The import fails, the hook program crashes, and it reports exit code 1 — which is neither “allow” (0) nor “block” (2). Claude Code shrugs and moves on. No error dialog. No warning.

🕳️
Gates that vanish without a sound

The user would keep working, believing the honesty gates are on duty — while every single one of them is absent. A broken feature that announces itself gets fixed; a broken safety net gets discovered the day you fall.

💡
Key insight: a bet is fine — an unchecked bet isn’t

Engineers call an assumption the whole design leans on an invariant. The fix for a risky invariant is rarely to redesign — it’s to write a check that fails loudly the day the assumption breaks. That’s exactly what Module 2 builds.

Module 2

An Install in a Bottle

You can’t unit-test a marketplace you don’t control. But you can build a perfect replica of what it does. TASK-39’s answer is one new test file — test/install-path.test.mjs — that performs a complete install inside a throwaway folder, then treats the result exactly the way Claude Code would.

This kind of test is called e2e, end-to-end. First step: simulate the install itself.

CODE — test/install-path.test.mjs 

function installPlugin({ name, source }) {
  const dest = join(mkdtempSync(join(tmpdir(), `praxisflux-install-${name}-`)), name);
  cpSync(join(repo, source), dest, { recursive: true });
  const lib = join(dest, "lib");
  if (existsSync(lib) && lstatSync(lib).isSymbolicLink()) {
    rmSync(lib);
    cpSync(join(repo, "lib"), lib, { recursive: true });
  }
  return dest;
}
PLAIN ENGLISH

“Install” one plugin, given its name and where it lives in the repository.

Make a fresh throwaway folder to install into — our bottle.

Copy the whole plugin folder over, exactly as the marketplace would.

Now find the copied “lib” entry — the recipe-card signpost from Module 1.

If it’s still a signpost rather than a real folder…

…tear the signpost out…

…and paste in a real copy of the shared code — page 12, copied onto the card.

Done with the special case.

Hand back the installed plugin’s location.

That’s the entire install, replicated in nine lines.

The test then asserts something blunt: no signpost of any kind survives in the installed copy, and the shared code is physically there.

Run it the way Claude Code runs it

Here’s the subtle part. Older tests imported the gate functions directly into the test program — like phoning the doorman at home to ask what he would say. This test instead walks up to the actual door:

1
Read the plugin’s own wiring

The command comes out of the installed plugin’s hooks.json — the same file Claude Code reads — not out of the test author’s memory of it.

2
Launch it as a real separate program

The test spawns the command through bash, exactly as the Stop event would.

3
Feed it a fake Stop event

Claude Code hands hooks a small JSON message on stdin; the test forges one, pointing at a fixture project.

4
Judge only the exit code

Whatever number the doorman reports — 0, 1, or 2 — is the verdict the test grades. Same contract as production, nothing mocked.

🎯
Why this matters for steering AI

When you ask an AI to “add tests,” it will usually take the phone-call shortcut — importing functions directly. If what you actually fear is the wiring (config files, launch commands, environment), say so: “spawn the real command like production does.” That one sentence is the difference between this test and a comforting illusion.

The smoke-detector principle

A silent smoke detector means one of two things: clean air, or a dead battery. From the silence alone, you cannot tell which. The only honest check is to make some smoke and demand a scream.

So every plugin’s hook faces three fixtures, and must produce three different verdicts:

🌤️
Clean air → exit 0

An empty project. The hook must run to completion and allow the stop — this alone proves the whole chain loads from the installed copy (a broken signpost would crash with exit 1 right here).

🔥
Smoke → exit 2, with the complaint on stderr

A project deliberately violating that plugin’s one rule. The gate must actually fire and say why — proving the pass wasn’t an empty formality.

🔁
Second alarm → exit 0 again

Same smoky project, but the fake event says “I already blocked you once.” The hook must yield — otherwise the assistant gets trapped in an endless argue-with-the-doorman loop.

💡
Key insight: test both verdicts, always

A test that only checks the happy path can pass vacuously — green because nothing happened, not because the right thing happened. Engineers guard against it by forcing the failure they claim to detect. If your AI-written tests never construct a failure, they may be measuring silence, not safety.

Each of the four hook-shipping plugins gets its own bespoke “smoke”: a lesson claiming progress with no files, a research page loading code from the internet, a task marked Done over an unproven spec, an abandoned review left in flight. One rule each — violated precisely.

Proving the test itself has teeth

One more turn of the screw: how do you know the test isn’t the dead battery? During the task, the author re-ran the install simulation with the dereference step deleted — the exact bug the test exists to catch. Result: the hook crashed with exit code 1, and the test failed loudly. The alarm screams when there’s real smoke.

The whole file now runs two ways on every pull request: inside the main test suite, and as its own separately-named job in CI, so an install-path regression is never buried inside a generic “tests failed.”

🧯
Good to know: coverage that recruits the future

The test derives its plugin list from the marketplace catalog. A future plugin that ships a Stop hook is pulled into coverage automatically — and the test fails with instructions if that plugin has no “smoke” fixture yet. The invariant stays checked without anyone remembering to extend the test.

Module 3

Honest Signs on the Door

The test proved the machinery. The last two acceptance criteria were about something softer and just as load-bearing: the signs on the door. A README is a set of claims, and this one had drifted from reality.

The install section still warned that you’d need special access “while this repo is private.” TASK-39 checked the facts on the day of the work: the repository had gone public, and the companion package was live on npm. The caveat wasn’t cautious; it was wrong — in the too-pessimistic direction, which still costs adopters who believe it.

🔍
Good to know: verify, don’t transcribe

The task’s brief assumed the repo was private and asked for the caveat to be kept. The work started by checking — one command against the live service — and the answer flipped what the README should say. Docs written from memory drift; docs written from checks don’t.

Three marked trails in

The rewritten install section reads like a good trailhead sign: every route that actually exists, each with what it’s for — and nothing that doesn’t exist.

🛍️

The marketplace

The primary surface: add the marketplace inside Claude Code and install plugins one by one — each arrives with its Stop hooks, on the very path Module 2’s test now proves.

📦

Per-plugin zips

Every release attaches one zip archive per plugin, packaged with the signpost already turned into real files — for installs that can’t reach the marketplace.

⚖️

Gates only, no install

Any project’s CI can run just the honesty gates — via a ready-made GitHub Action or a single npx command — without adopting the plugins at all.

And one sentence above all three, now true and checkable: every surface is public; no special access required.

The doorman is written in bash

One claim was still missing from the door: which machines does any of this work on? The answer hides in the doorman itself. Here is the working half of gate.sh, verbatim:

CODE — team-review/scripts/gate.sh 

here="$(cd "$(dirname "$0")" && pwd)"
root="${CLAUDE_PLUGIN_ROOT:-$(dirname "$here")}"
node_bin="$(command -v node 2>/dev/null)"
if [ -z "$node_bin" ]; then
  node_bin="$(${SHELL:-/bin/sh} -lc 'command -v node' 2>/dev/null)"
fi
if [ -z "$node_bin" ]; then
  exit 0
fi
exec "$node_bin" "${root}/scripts/stop.mjs"
PLAIN ENGLISH

Figure out which folder this script itself lives in.

The plugin’s root: use the location Claude Code provides, or fall back to “one folder up from me.”

Look for Node on this machine, hiding any error message.

If it wasn’t found…

…try once more the way the user’s own login shell would find it.

End of the retry.

Still nothing?

Then bow out quietly — a missing runtime must never block the assistant from finishing.

End of the give-up check.

Node found: hand the whole job over to the real gate program.

Every line of that is bash — and Windows doesn’t ship bash. Add that Windows checkouts don’t reliably support symlinks, and a native-Windows install would recreate Module 1’s nightmare exactly: gates silently absent.

Saying “no” out loud

There were two honest options: do the large engineering to support Windows natively, or declare it out of scope — loudly, with reasons. The owner chose the declaration. The README now states: macOS and Linux supported; native Windows out of scope because the shims are bash and the symlinks assume core.symlinks; Windows users should run under WSL, where everything behaves as on Linux.

And one more sentence with teeth: revisiting the decision means a tested PowerShell shim path, not a doc edit — the claim can only be upgraded by new evidence, never by wishful rewording.

💡
Key insight: a recorded “no” beats a silent gap

Silence about a platform reads as “probably fine” — and fails as a mystery on the user’s machine, eroding trust in everything else. A stated limitation with a reason and an escape hatch costs one paragraph and fails as a known, navigable fact. In your own projects: never let scope be implied by omission.
