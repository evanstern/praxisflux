Module 2 · Three Runs and a Cat

The line, proven live

A pipeline diagram proves nothing. TASK-22 ran the line three times against three different situations — including one where the AI wrote real code that now lives on a real project's main branch.

1
The lie detector

A project whose board claimed Done over unfinished work. Gate FAILED in writing → the AI repaired it from the gate’s own words ($0.61) → gate PASSED → a human approved the landing.

2
The cat

A real mini-project: a terminal tamagotchi. The AI station implemented its first feature from the spec — state, hatching, mood faces — on a fresh branch ($1.38). Gate passed; a human met the cat, then approved the merge.

3
The promotion

A fully-proven spec with a lagging board. The pipeline promoted the task to Done — derived from the files, never hand-set — and the human approved that landing too.

The approval window earned its keep

In run 2, the human inspection before approving caught something no gate checks: the pet’s save file — runtime data created just by playing with it — was about to ride into the merge, because the landing step commits everything in the folder. One quick fix later, save files are permanently ignored.

⚠️
The boundary, stated plainly

The gate proves the bookkeeping is honest. It does not review the work. That review is exactly what the human station is for — automation that skips it isn't a pipeline, it’s a liability.

Key Insight

Nothing in the underlying system knows n8n exists. Because state lives in files and checkers judge files, the whole flow ported to an orchestrator with one small connector service — swapping orchestrators is a plumbing decision, not a redesign.

In run 2, the gate passed — yet the human still found a problem before approving. What does that show?
