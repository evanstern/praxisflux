Anatomy of a Plugin

The program is written in English — open the box and see what's really inside.

The main program is… a document?

Open the plugin called educate and look for its engine. You'd expect thousands of lines of JavaScript. You'll find something much stranger.

🤔

What you expect

A mountain of machinery — thousands of lines of code doing the teaching, tracking, and checking.

📜

What you find

SKILL.md — a 133-line English document. That is the main program. Claude Code reads it and follows it, step by step.

The prose isn't documentation about the program — the prose is the program. The JavaScript that ships alongside it is tiny, and it mostly exists to check the AI's work, not to do the work.

💡
Prose is source code now

In AI-native software, carefully engineered English is source code: kept under version control, reviewed like code, even tested like code. When you write instructions for an AI — a custom prompt, a workflow — you are programming.

Unboxing a plugin

Think of praxis as a cordless power-tool platform: one battery system powers many different tools. Each plugin is one tool's box — and every box is packed the same way: a label, the manuals, an inspector's checklist, and some starter jigs. Here's the real folder tree of educate:

educate/ the whole tool box
.claude-plugin/plugin.json the label on the box — the plugin's manifest
hooks/hooks.json "run my inspector when Claude finishes" — a hook (more in Module 3)
skills/ the manuals Claude follows
lesson/SKILL.md the main manual — 133 lines of English
start/SKILL.md the setup manual — 84 lines
gates/ the inspector's checklist — stars in Module 3
dod.mjs a gate: "is this lesson really done?"
wiki.mjs a gate: "are the notes still fresh?"
scripts/ gate.sh · stop.mjs · progress.mjs · wiki.mjs — the wiring that runs the inspector
templates/ starter files it plants in your project
CLAUDE.md · progress.schema.json standing rules + a progress ledger, ready to copy
.template/ checklist.md · raw-notes.md — blank template pages for each lesson

One quirk worth knowing: plugins have no "always-on" slot — their skills load lazily, only when summoned. So a plugin that needs standing rules plants a CLAUDE.md into your project folder. That's exactly what templates/CLAUDE.md is for.

Now let's read the label on the box — plugin.json, line by line:

CODE — educate/.claude-plugin/plugin.json 

{
  "name": "educate",
  "version": "0.1.0",
  "description": "Turn a folder into a Socratic learning project: teach, author a build SPEC, and gate each lesson done on auditable artifacts.",
  "author": { "name": "praxis maintainers" },
  "keywords": ["learning", "teach-me", "socratic", "curriculum", "orchestration"]
}
PLAIN ENGLISH

The label starts — this whole file is one JSON object.

The plugin's name: what you'd type to install it from the marketplace.

A version number, so updates can be tracked like any other software.

One honest sentence about what's inside the box.

Who made it — a name and an email, right on the label.

Search keywords, so the right person can find this tool in the catalog.

…and the label ends. That's the entire manifest.

A skill up close: one sentence, three jobs

Every skill manual opens with frontmatter — a tiny YAML card stapled to page one. Here's the real one from the build plugin's implement skill, the manual for turning a SPEC into working code:

CODE — build/skills/implement/SKILL.md 

---
name: implement
description: Implement a build SPEC handed off from a learning lesson (or any producer), verify it works, and return findings for the lesson to fold back in. Use when a lesson has handed off a SPEC to build, when the user says "run build", "/build-me", "build the spec", or points at a pending handoff request. This plugin OWNS the implementation leg — educate teaches and authors the SPEC; build implements it and returns what it learned.
---
PLAIN ENGLISH

Three dashes open the frontmatter card.

The skill's name — what this manual is called.

One long sentence doing three jobs at once: it explains what the skill does, lists the exact phrases that wake it up ("run build", "/build-me"), and stakes out turf — "this plugin OWNS the implementation leg," so no other plugin tries to do this work.

Three dashes close the card; everything below is the manual itself.

That description line is the densest engineering in the whole plugin. Unpack it:

📖

Documentation

A human skimming the folder learns what the skill does from one sentence.

🔔

The doorbell

Claude matches your words against it — say "run build" and this manual gets opened.

🚩

A turf declaration

"This plugin OWNS the implementation leg" — so educate teaches, build builds, and they never collide.

The shared battery: lib/

Back to the power-tool metaphor: five different tools, one battery system. The battery is lib/ — nine tiny modules of JavaScript that every plugin snaps onto. A few of the cells:

🧭
project-root

Find where the project lives, from any folder you're standing in.

🚦
lifecycle

Enforces "status can't exceed proof" — the rulebook behind Module 3's gates.

📮
handoff

The mail system plugins use to pass work to each other — Module 4's whole story.

🌱
installer

Plants starter files into your project honestly — never overwriting your edits.

The whole battery is 455 lines across nine files — none over 81 lines — and it's zero-dependency: it uses nothing but what ships with Node.js, not a single downloaded dependency. And at packaging time the whole lib/ folder is copied into each plugin, so every installed plugin is self-contained — Module 5 owns that story.

🔋
455 lines run five plugins

Infrastructure doesn't have to be huge — every one of the nine shared files is smaller than this page's HTML.

How a plugin finds home

You're five folders deep in a project and ask Claude to "resume the lesson." How does educate know where the project's home is? The same way git finds your repo from any subfolder: it looks for a landmark. Praxis has two placement models:

⬆️

educate walks UP

From wherever you stand, it climbs parent folders until one contains a topics/ landmark. That folder is home.

⬇️

research walks DOWN

It sweeps downward, collecting every folder marked with a .research-vault file — many homes, not one.

The landmark trick is two tiny functions. First, a factory that builds marker checkers:

CODE — lib/project-root.mjs 

/** A marker function: presence of a child file/dir named `name`. */
export function hasChild(name) {
  return (dir) => existsSync(join(dir, name));
}
PLAIN ENGLISH

A note from the author: this manufactures little landmark-checkers.

Give it a name to look for — say, "topics" — and it builds you a custom checker.

The checker answers one yes/no question: does this folder contain something with that name?

That's the whole factory.

Then the climber that carries a checker up the folder tree:

CODE — lib/project-root.mjs 

export function findRootUpwards(startDir, markerFn) {
  let dir = resolve(startDir);
  const { root: fsRoot } = parse(dir);
  for (;;) {
    if (markerFn(dir)) return dir;
    if (dir === fsRoot) return null;
    dir = dirname(dir);
  }
}
PLAIN ENGLISH

Start from any folder, carrying one of those landmark checkers along.

Turn "wherever we're standing" into a full, unambiguous address.

Note where the top of the disk is — the root, the point where there's no more "up".

Now loop, forever if needed…

Is the landmark here? Then this folder is home — done.

Hit the top of the disk without finding it? Give up and report "no home".

Otherwise, step up into the parent folder…

…and go around the loop again from there.

End of the climber.

That's it. hasChild("topics") builds the question; findRootUpwards keeps asking it, one parent folder at a time, until it finds home or runs out of "up".
