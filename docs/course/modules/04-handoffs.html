Handoffs: Interoffice Mail & the Signed Logbook

How two plugins that never meet pass work through a mailbox folder — and why the proof lives somewhere else entirely.

Two programs that have never met

In Module 1 you watched educate hand a SPEC to build. But here's the rule that makes praxis praxis: plugins never call each other. Ever.

So how does a work order travel between two departments that don't share a phone line? The answer is almost embarrassingly simple: it's a text file, dropped in a mailbox folder. That's a handoff.

Here's where everything in this module lives. Two separate worlds: the mailbox, and the records office.

.handoff/ the mailbox — gitignored, invisible to git

Keep those two worlds apart in your head — the envelopes and the logbook. The entire module is the story of why they're separate.

The envelope

Sending a message = writing one Markdown file. The address label on the front is called frontmatter; everything under it is the payload.

praxis mail
.handoff/
id:    spec-101
kind:  request (or response)
from:  educate
to:    build
ref:   — "in reply to memo #…" — the ref
The body — the payload. The mail room never opens it; only educate and build understand what's written inside.

And here is the entire mail system — one function, from lib/handoff.mjs. Eight lines.

CODE — lib/handoff.mjs 

export function writeHandoff(root, { id, kind, from, to, ref = "", title = "", body = "" }) {
  if (!id || !kind || !from || !to) throw new Error("handoff requires id, kind, from, to");
  const dir = ensureHandoffDir(root);
  const fm = [`id: ${id}`, `kind: ${kind}`, `from: ${from}`, `to: ${to}`, `ref: ${ref}`, `title: ${title}`].join("\n");
  const path = join(dir, `${id}.md`);
  writeFileSync(path, `---\n${fm}\n---\n${body}\n`);
  return path;
}
PLAIN ENGLISH

The send function. Every envelope arrives with an id, a kind, a from, a to — plus an optional ref, title, and body.

No complete address label? Refuse the envelope on the spot — undeliverable mail helps no one.

Make sure the mailbox folder (.handoff/) exists before dropping anything in.

Write the address label: six plain-text lines — id, kind, from, to, ref, title — stacked one per line.

The envelope's filename is its id: spec-101 becomes spec-101.md.

Staple the label on top (between --- fences) and drop the whole letter into the mailbox.

Hand back the path — "your mail is at…"

That's it. The entire postal service.

Notice what's not here: the mail room never reads the body. The payload is opaque — only the sender and receiver understand it.

The human is part of the protocol

No plugin ever triggers another. Each one finishes, stamps its logbook, and tells you what to run next. The loop's invisible participant is the user.

Reading your mail — archived, never shredded

When build read spec-101.md, the envelope didn't vanish. Consuming a message moves it to the answered drawer, .handoff/consumed/ — so you can always do archaeology later.

CODE — lib/handoff.mjs 

export function markConsumed(root, id) {
  const src = join(handoffDir(root), `${id}.md`);
  if (!existsSync(src)) return false;
  const cdir = join(handoffDir(root), "consumed");
  mkdirSync(cdir, { recursive: true });
  renameSync(src, join(cdir, `${id}.md`));
  return true;
}
PLAIN ENGLISH

Reading your mail, step one: find the envelope by its id.

Here's where it should be sitting in the mailbox.

No envelope there? Say so and stop — nothing to consume.

The "answered" drawer lives inside the mailbox: .handoff/consumed/.

Build the drawer if this is the first answered letter.

Move — never shred — the envelope into the drawer.

Report success: the mail is archived.

Notice what's missing: no receipt. Writing the logbook entry is the recipient's job — the mail room never writes in anyone's books.

One more thing about the mailbox: it's invisible on purpose. Run git status mid-handoff and the envelopes simply aren't there. That invisibility is contractual — there's a unit test for it.

CODE — test/handoff.test.mjs 

  // .handoff/ is gitignored so it never pollutes git status
  assert.match(readFileSync(join(root, ".gitignore"), "utf8"), /^\.handoff\/$/m);
PLAIN ENGLISH

The promise, written down: the mailbox never clutters git's view of the project.

And a robot check that the promise holds — the .gitignore file must contain exactly the line ".handoff/", or this test fails the build.

The repo's own documentation states the split in one breath — payloads over here, evidence over there.

docs/handoff-protocol.md 
A handoff message is *plumbing*, not a work product — it must not clutter `git status`. So:

- **Payloads** live in a **gitignored `.handoff/`** at the project root (the installer adds the
  ignore). They are opaque markdown files with a small envelope; consuming one moves it to
  `.handoff/consumed/`.
- **Evidence** that a handoff happened — and what it changed — is recorded in the consumer's own
  **tracked state** (e.g. educate's `progress.json`). Gates read the evidence, never the loose
  payload files.

The double lock — no rubber stamps

Remember Module 3's cliffhanger — the gate failure about handoff.foldedIn? This is that story. The return leg — folding build's findings back into the lesson — is the step an AI most wants to skip.

So finishing a lesson requires two locks to open, not one.

🔒

Lock 1 — the flag

progress.json says handoff.foldedIn: true. Easy to stamp — which means easy to rubber-stamp.

🔒

Lock 2 — the residue

A real ## Post-build section written into guide.md or raw-notes.md — physical residue a gate can read.

Here's the rubber-stamp detector itself, from educate's Definition-of-Done gate. It fires precisely when the flag is set but the disk is empty.

CODE — educate/gates/dod.mjs 

} else if (lesson.status === "done" && h.foldedIn && !hasReturnLegResidue(lessonDir)) {
  problems.push(`${lesson.id}: status=done with handoff.foldedIn=true but no durable return-leg residue on disk — add a "## Post-build" section to guide.md or raw-notes.md; the return leg can't be rubber-stamped`);
PLAIN ENGLISH

The trap: status claims done, the logbook flag swears the findings were folded in — but the disk scan finds no real Post-build section anywhere…

…then block the lesson and say exactly how to fix it: write the section into a tracked file. A checkbox can't be checked into truth.

💡
Transient payload, durable evidence

The messages are disposable plumbing hidden from git; the proof lives in tracked state that gates can check. This double lock — flag + residue — is how you stop any AI from rubber-stamping the step it most wants to skip.
