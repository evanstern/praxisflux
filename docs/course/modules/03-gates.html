Gates — The Inspector Who Can’t Be Sweet-Talked

How a tiny, stubborn script stops Claude from saying “done” when it isn’t

The “done!” problem

AI assistants have a bad habit: declaring “done!” when it isn’t. If you’ve ever vibe-coded, you’ve lived this — a confident summary, a little victory lap… and the thing doesn’t actually work.

Praxis’s answer is a piece of code so simple it can’t be argued with. Here’s the exact situation it exists to catch:

💬 What Claude says

“Lesson complete! Deck written, notes captured, everything wrapped up. Marking it done. ✨”

💾 What the disk says
✓ checklist.md
✓ raw-notes.md
✗ deck.html — missing

A status can’t exceed the artifacts that prove it.

Praxis enforces that motto with a gate. The metaphor to keep in your head is a building inspector and the certificate of occupancy:

👷

The builder — Claude

Swears the wiring is done. Might even believe it. Confidence is not evidence.

🕵️

The inspector — the gate

Physically walks the site and checks the actual walls. Doesn’t negotiate, doesn’t care how confident the builder sounds.

📋

The fixed checklist

The list of required evidence, written down in advance. Not up for discussion at inspection time.

🏠

The certificate

Permission to call the job finished. The builder can’t print their own — only the inspector issues it.

And the inspector shows up automatically, at the end of every job. Separate inspectors — one per installed plugin — each check their own specialty on the same house.

The claim vs. the disk

educate — the teaching plugin from Module 2 — tracks every lesson’s progress on a ladder. Each rung upward must be earned with an artifact: a real file on disk that proves the work happened.

So how does code check a claim? It doesn’t read Claude’s summary. It asks the disk. This is a tiny function from the shared chassis, and it’s the heart of the whole module:

lib/lifecycle.mjs 

/** Derive { key: bool } for each artifact by checking its filename on disk in `dir`. */
export function computeArtifacts(dir, artifactFiles) {
  const out = {};
  for (const [key, file] of Object.entries(artifactFiles)) out[key] = existsSync(join(dir, file));
  return out;
}
PLAIN ENGLISH

The whole recipe in one sentence: for each expected artifact, check whether its file is really on the disk.

A reusable function anyone can call: hand it a folder and the list of expected evidence files…

…start with an empty scorecard…

…then for every claim (“there’s a deck”, “there are notes”), ask the disk: does that exact file exist in that folder? Write down yes or no.

Hand back the scorecard.

That’s the entire trick.

Don’t believe the label — look at the disk. Truth is derived from what’s observable, never from what’s declared. And the course motto isn’t just a slogan; it sits in the real code as a comment:

educate/gates/dod.mjs 

// a status may not exceed the artifacts that prove it

The motto, verbatim, where it does its job.

Compare the lesson’s claimed rung against the rungs the disk can prove — and every gap becomes a named problem on a list.

The moment of truth: the Stop hook

A checklist is useless if nobody dispatches the inspector. Here’s the beautiful part: Claude Code lets you register programs that run the instant Claude tries to finish. That slot is called a Stop hook (a hook that fires at the Stop moment).

Registering one takes zero code — it’s pure configuration, a few lines of JSON:

educate/hooks/hooks.json 

"hooks": {
    "Stop": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/gate.sh"
          }
        ]
      }
    ]
  }
PLAIN ENGLISH

Dear Claude Code: here are programs I want you to run for me…

…specifically at the Stop moment — the instant Claude tries to finish up.

Here comes one rule.

Which situations does it apply to? The star means all of them — no exceptions, ever.

And here’s what to run when it fires…

…a single entry…

…it’s a command to execute…

…namely: run the gate script that ships inside this plugin, wherever the plugin happens to be installed.

End of the entry.

…close the list of things to run…

…close the rule…

…close the Stop section…

…and done. Notice: not one line of logic. The whole registration is configuration.

That command line says bash, which just means “run this script.” But how does a script answer the question “may Claude stop?” With the one thing every program produces as it ends: an exit code.

0
All fine

The program ends happy. Claude is allowed to stop.

2
Block

Claude may not stop — and whatever the script printed becomes the message Claude reads next.

The full deal between Claude Code and the gate fits in two lines. The gate receives a note through stdin and writes any complaints to stderr:

lib/gate-runner.mjs 

// Contract (Claude Code Stop hook): stdin is JSON with { stop_hook_active, cwd, … };
// exit 0 = allow the model to stop; exit 2 = block, and stderr becomes the message it sees.
PLAIN ENGLISH

Incoming mail: a small JSON note with a flag saying whether a gate already blocked once, plus the folder Claude was working in.

Outgoing answer: just a number. 0 — free to go. 2 — blocked, and every word printed to the error channel lands in front of Claude as its new to-do list.

The verdict, in code

The shared engine in lib/ delivers the verdict for every plugin the same way. This is the courtroom moment:

lib/gate-runner.mjs 

const { block, message, warnings } = evaluate(input, gates);
if (block) {
  process.stderr.write([message, warnings].filter(Boolean).join("\n") + "\n");
  exit(2);
} else {
  if (warnings) process.stderr.write(warnings + "\n");
  exit(0);
}
PLAIN ENGLISH

Run every check and collect three things: should we block, the message explaining why, and any softer warnings.

If something required is missing…

…print the full problem list on the error channel — the exact text Claude Code will hand back to Claude…

…and end with the number 2: permission to stop is denied.

Otherwise…

…gentle reminders still get printed, but they ride along without blocking anything…

…and the number 0 says: all clear, Claude may finish.

Court adjourned.

And here’s educate declaring its own inspector. A gate is just three answers — am I relevant here? what’s broken? what’s merely worth mentioning? — handed to that shared engine:

educate/scripts/stop.mjs 

const educateGate = {
  name: "educate",
  resolveRoots: (startDir) => {
    const root = findRootUpwards(startDir, hasChild("topics"));
    return root ? [root] : [];
  },
  check: (root) => gateProblemsForProject(root),
  // Corpus-index freshness is a reminder, not a DoD violation — warn (never block) if a WIKI.md
  // has drifted from the vaults on disk. Run wiki.mjs --sync (or progress.mjs --sync) to refresh.
  warn: (root) => wikiStalenessWarnings(root),
};

runStopHook({ gates: [educateGate] });
PLAIN ENGLISH

educate describes its inspector as a small bundle of answers.

The name on the inspector’s badge.

Answer one: am I even relevant here?

Walk up the folders from where Claude was working, looking for one that contains topics/ — the exact same findRootUpwards trick from Module 2.

Found one? That project is my jurisdiction. Found nothing? Hand back an empty list and stay out of it.

End of answer one.

Answer two: what’s actually broken? Run the definition-of-done checks on that project.

A note from the authors: the next check is deliberately gentle…

…a stale index is worth mentioning, never worth blocking over.

Answer three: what’s merely worth mentioning?

The inspector is fully described.

…a breath…

Hand it to the shared engine — the chassis does the intercepting, the reading, and the exiting, identically for every plugin.

💡
The gate can’t be sweet-talked

The enforcement works precisely because the enforcer is dumber than the thing it checks. A deterministic file check returns the same verdict no matter how confident the summary sounds — existsSync(deckPath) has no vibes. This is the most stealable pattern in praxis: never trust an AI’s “done” — demand checkable evidence, and check it with dumb code.

🚧
Fails closed

What if the inspector trips on the stairs — the gate itself crashes? The crash is reported as a problem and Claude still can’t stop. A broken inspector halts the line rather than waving everything through.

Three more quiet safeguards are hiding in that little bundle:

🔁
No infinite argument

Remember stop_hook_active in the stdin note? It tells the gate “you already blocked once” — so Claude can’t get trapped in an endless loop of blocked stops.

🤫
Silent when off-duty

No topics/ folder anywhere above? The gate resolves zero projects and becomes a no-op. That’s why installed plugins’ inspectors stack additively: each speaks only on its own beat.

👀
Look, don’t touch

The gate is read-only. It inspects the disk and reports — it never “fixes” anything itself. Fixing is the builder’s job.

So what does a blocked Claude actually read? A real one, verbatim:

[git-basics] 002-branching: status=done but the return leg is unrecorded (handoff.foldedIn=false) — fold the build findings back into the lesson first

That word handoff? A lesson isn’t done until the work it mailed to another plugin has come back and been folded in. How that mail gets sent — and why the inspector demands the return receipt — is Module 4’s whole story. 📬
