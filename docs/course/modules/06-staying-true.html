Sailors navigate by nautical charts, and every chart carries a stamp: surveyed as of… A chart isn't wrong because it's old — it's wrong when the coastline actually changed since the survey. Notes about code age exactly the same way.

A confession first: docs rot in this very repo. And conveniently, git — the ledger recording every change ever made here — can answer "did the coastline move?" precisely.

📜 True story, this repo

Three plugin READMEs in praxis still say "scaffold — not yet implemented" while their skills are fully written and working. The docs went stale — which is exactly the disease this module's gate cures.

The patient is grounding-wiki (from Module 1): it reads a codebase and writes one note per concept into docs/wiki/ — a corpus other plugins trust instead of re-reading the raw code. Trusted documentation is only useful if it stays true.

The pin: every note declares what it looked at

Every chart carries its survey stamp; every wiki note carries a pin. It lives in the note's frontmatter, written in YAML. Here's the template every note starts from — real file, real words.

grounding-wiki/templates/note.md 

---
name: <kebab-case, matches filename>
description: <one line — used to decide relevance during recall>
kind: component | concept | pipeline | pattern
sources:
  - <repo-relative path whose change invalidates this note>
verified_against: <full commit hash>
---
Plain English

The label block opens — three dashes fence the frontmatter.

The note's name, written like-this so it matches its own filename.

One line saying what the note covers, so a search can decide if it's relevant.

What kind of thing it describes: a component, an idea, a pipeline, or a pattern.

Now the important part — the list of files this note is describing…

…each one a path "whose change invalidates this note" — the template's own words.

The pin itself: the exact snapshot of the project this note was checked against.

…and the label block closes.

Two fields do all the work. sources: is the note's coastline — the exact files it describes; if one changes, the note is invalidated. verified_against: is the survey stamp — the commit it was checked against, named by its commit hash.

sources: The coastline — which files this note describes
verified_against: The survey stamp — which snapshot it was last checked against

Asking git: did the coastline move?

With a pin in every note, a dumb check can put one precise question to the ledger. The whole freshness gate boils down to a single git log question:

git log <pin>..HEAD -- <sources> "List every change to these files since that snapshot." HEAD means "now."

An empty answer means fresh. Any answer means stale — and the guilty commits get named. Here's the real check, from the real gate.

grounding-wiki/gates/freshness.mjs 

let changed = "";
try {
  changed = git(repoRoot, ["log", "--oneline", `${pin}..HEAD`, "--", ...sources]);
} catch (e) {
  fails.push(`${rel}: git log failed for its sources (${String(e).split("\n")[0]})`);
  continue;
}
if (changed) {
  const lines = changed.split("\n");
  fails.push(`${rel}: STALE — sources changed since ${pin.slice(0, 12)} (${lines.length} commit(s), e.g. ${lines[0]})`);
}
Plain English

Start with an empty answer sheet.

Try asking the question — carefully, in case git can't answer at all…

…the question: "list every change to this note's files since its pinned snapshot."

If the asking itself broke down…

…record that this note couldn't be checked, quoting the error…

…and move on to the next note.

Done with the careful part.

If git came back with anything at all…

…count how many changes it found…

…and flag the note STALE — naming the snapshot, the count, and the first guilty commit.

Verdict delivered for this note.

What a failure actually looks like
docs/wiki/alpha.md: STALE — sources changed since abc123def456 (3 commit(s), e.g. abc1234 fix parser)
🧭
Fresh means "nothing it describes has changed" — not "correct"

The gate proves the coastline hasn't moved since the survey; it can't prove the survey was accurate. And a note with no sources: at all earns a warning, not a failure — its staleness is unverifiable, which is its own kind of suspicious.

Stale isn't fatal — it's a repair ticket

A stale note is presumed lying until re-verified. The wiki-update skill is the repair loop, and the failure message already told it where to look:

1
Find the stale notes

Run the freshness gate — it names each stale note and the guilty commits.

2
Re-verify against the actual diff

Read the diff — what really changed in the sources — and fix whatever the note now gets wrong.

3
Re-pin

Stamp the note with the new snapshot in verified_against:. Surveyed as of… today.

📄 alpha.md · sources: src/a.txt

One commit touches src/a.txt → STALE

📄 beta.md · sources: src/b.txt

Its file untouched → FRESH

That exact flip is proven by a real automated test in the repo (test/grounding-wiki.freshness.test.mjs). And notice when this gate runs: not on every Stop hook — cheap universal checks ride the Stop; contextual ones like freshness run at milestones, when a wiki or a course is being built or updated.

Steal these five

None of this is praxis-specific. These five principles travel — take them to every AI project you ever build.

🚦

Status can't exceed proven artifacts

Make your AI produce checkable evidence, then verify with dumb code — not vibes. M3

📁

Compose through files, not calls

Phases that read and write files can be run, skipped, replaced, and audited — and work with any tool, including a human. M1 · M4

🧾

Flags aren't proof — demand residue

Double-lock the step the AI most wants to skip: a flag can be flipped, real residue can't be faked cheaply. M4

📦

Share the source, vendor the artifact, robot-check the copies

Duplication is fine when a machine verifies it. M5

📌

Pin every claim to what it looked at

"Verified against snapshot X" turns "trust me" into a checkable statement. M6
