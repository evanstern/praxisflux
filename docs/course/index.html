How a plugin finds home

You're five folders deep in a project and ask Claude to "resume the lesson." How does educate know where the project's home is? The same way git finds your repo from any subfolder: it looks for a landmark. Praxis has two placement models:

⬆️

educate walks UP

From wherever you stand, it climbs parent folders until one contains a topics/ landmark. That folder is home.

⬇️

research walks DOWN

It sweeps downward, collecting every folder marked with a .research-vault file — many homes, not one.

The landmark trick is two tiny functions. First, a factory that builds marker checkers:

CODE — lib/project-root.mjs 

/** A marker function: presence of a child file/dir named `name`. */
export function hasChild(name) {
  return (dir) => existsSync(join(dir, name));
}
PLAIN ENGLISH

A note from the author: this manufactures little landmark-checkers.

Give it a name to look for — say, "topics" — and it builds you a custom checker.

The checker answers one yes/no question: does this folder contain something with that name?

That's the whole factory.

Then the climber that carries a checker up the folder tree:

CODE — lib/project-root.mjs 

export function findRootUpwards(startDir, markerFn) {
  let dir = resolve(startDir);
  const { root: fsRoot } = parse(dir);
  for (;;) {
    if (markerFn(dir)) return dir;
    if (dir === fsRoot) return null;
    dir = dirname(dir);
  }
}
PLAIN ENGLISH

Start from any folder, carrying one of those landmark checkers along.

Turn "wherever we're standing" into a full, unambiguous address.

Note where the top of the disk is — the root, the point where there's no more "up".

Now loop, forever if needed…

Is the landmark here? Then this folder is home — done.

Hit the top of the disk without finding it? Give up and report "no home".

Otherwise, step up into the parent folder…

…and go around the loop again from there.

End of the climber.

That's it. hasChild("topics") builds the question; findRootUpwards keeps asking it, one parent folder at a time, until it finds home or runs out of "up".

03

Gates — The Inspector Who Can’t Be Sweet-Talked

How a tiny, stubborn script stops Claude from saying “done” when it isn’t

The “done!” problem

AI assistants have a bad habit: declaring “done!” when it isn’t. If you’ve ever vibe-coded, you’ve lived this — a confident summary, a little victory lap… and the thing doesn’t actually work.

Praxis’s answer is a piece of code so simple it can’t be argued with. Here’s the exact situation it exists to catch:

💬 What Claude says

“Lesson complete! Deck written, notes captured, everything wrapped up. Marking it done. ✨”

💾 What the disk says
✓ checklist.md
✓ raw-notes.md
✗ deck.html — missing

A status can’t exceed the artifacts that prove it.

Praxis enforces that motto with a gate. The metaphor to keep in your head is a building inspector and the certificate of occupancy:

👷

The builder — Claude

Swears the wiring is done. Might even believe it. Confidence is not evidence.

🕵️

The inspector — the gate

Physically walks the site and checks the actual walls. Doesn’t negotiate, doesn’t care how confident the builder sounds.

📋

The fixed checklist

The list of required evidence, written down in advance. Not up for discussion at inspection time.

🏠

The certificate

Permission to call the job finished. The builder can’t print their own — only the inspector issues it.

And the inspector shows up automatically, at the end of every job. Separate inspectors — one per installed plugin — each check their own specialty on the same house.

The claim vs. the disk

educate — the teaching plugin from Module 2 — tracks every lesson’s progress on a ladder. Each rung upward must be earned with an artifact: a real file on disk that proves the work happened.

The lesson ladder — climb only with receipts 🧗
planned the only free rung — just a row in the plan
scaffolded checklist.md
taught raw-notes.md
spec’d handoff evidence → Module 4
built handoff evidence → Module 4
decked deck.html
done 🏁 guide.md

So how does code check a claim? It doesn’t read Claude’s summary. It asks the disk. This is a tiny function from the shared chassis, and it’s the heart of the whole module:

lib/lifecycle.mjs 

/** Derive { key: bool } for each artifact by checking its filename on disk in `dir`. */
export function computeArtifacts(dir, artifactFiles) {
  const out = {};
  for (const [key, file] of Object.entries(artifactFiles)) out[key] = existsSync(join(dir, file));
  return out;
}
PLAIN ENGLISH

The whole recipe in one sentence: for each expected artifact, check whether its file is really on the disk.

A reusable function anyone can call: hand it a folder and the list of expected evidence files…

…start with an empty scorecard…

…then for every claim (“there’s a deck”, “there are notes”), ask the disk: does that exact file exist in that folder? Write down yes or no.

Hand back the scorecard.

That’s the entire trick.

Don’t believe the label — look at the disk. Truth is derived from what’s observable, never from what’s declared. And the course motto isn’t just a slogan; it sits in the real code as a comment:

educate/gates/dod.mjs 

// a status may not exceed the artifacts that prove it
for (const p of lifecycle.check(lessonDir, lesson.status)) problems.push(`${lesson.id}: ${p}`);
PLAIN ENGLISH

The motto, verbatim, where it does its job.

Compare the lesson’s claimed rung against the rungs the disk can prove — and every gap becomes a named problem on a list.

The moment of truth: the Stop hook

A checklist is useless if nobody dispatches the inspector. Here’s the beautiful part: Claude Code lets you register programs that run the instant Claude tries to finish. That slot is called a Stop hook (a hook that fires at the Stop moment).

Registering one takes zero code — it’s pure configuration, a few lines of JSON:

educate/hooks/hooks.json 

"hooks": {
    "Stop": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/gate.sh"
          }
        ]
      }
    ]
  }
PLAIN ENGLISH

Dear Claude Code: here are programs I want you to run for me…

…specifically at the Stop moment — the instant Claude tries to finish up.

Here comes one rule.

Which situations does it apply to? The star means all of them — no exceptions, ever.

And here’s what to run when it fires…

…a single entry…

…it’s a command to execute…

…namely: run the gate script that ships inside this plugin, wherever the plugin happens to be installed.

End of the entry.

…close the list of things to run…

…close the rule…

…close the Stop section…

…and done. Notice: not one line of logic. The whole registration is configuration.

That command line says bash, which just means “run this script.” But how does a script answer the question “may Claude stop?” With the one thing every program produces as it ends: an exit code.

0
All fine

The program ends happy. Claude is allowed to stop.

2
Block

Claude may not stop — and whatever the script printed becomes the message Claude reads next.

The full deal between Claude Code and the gate fits in two lines. The gate receives a note through stdin and writes any complaints to stderr:

lib/gate-runner.mjs 

// Contract (Claude Code Stop hook): stdin is JSON with { stop_hook_active, cwd, … };
// exit 0 = allow the model to stop; exit 2 = block, and stderr becomes the message it sees.
PLAIN ENGLISH

Incoming mail: a small JSON note with a flag saying whether a gate already blocked once, plus the folder Claude was working in.

Outgoing answer: just a number. 0 — free to go. 2 — blocked, and every word printed to the error channel lands in front of Claude as its new to-do list.

The verdict, in code

The shared engine in lib/ delivers the verdict for every plugin the same way. This is the courtroom moment:

const { block, message, warnings } = evaluate(input, gates);
if (block) {
  process.stderr.write([message, warnings].filter(Boolean).join("\n") + "\n");

Otherwise…

💡
The gate can’t be sweet-talked

The enforcement works precisely because the enforcer is dumber than the thing it checks. A deterministic file check returns the same verdict no matter how confident the summary sounds — existsSync(deckPath) has no vibes. This is the most stealable pattern in praxis: never trust an AI’s “done” — demand checkable evidence, and check it with dumb code.

🚧
Fails closed

What if the inspector trips on the stairs — the gate itself crashes? The crash is reported as a problem and Claude still can’t stop. A broken inspector halts the line rather than waving everything through.

Three more quiet safeguards are hiding in that little bundle:

🔁
No infinite argument

Remember stop_hook_active in the stdin note? It tells the gate “you already blocked once” — so Claude can’t get trapped in an endless loop of blocked stops.

🤫
Silent when off-duty

No topics/ folder anywhere above? The gate resolves zero projects and becomes a no-op. That’s why installed plugins’ inspectors stack additively: each speaks only on its own beat.

👀
Look, don’t touch

The gate is read-only. It inspects the disk and reports — it never “fixes” anything itself. Fixing is the builder’s job.

Keep those two worlds apart in your head — the envelopes and the logbook. The entire module is the story of why they're separate.

The envelope

Sending a message = writing one Markdown file. The address label on the front is called frontmatter; everything under it is the payload.

praxis mail
.handoff/
id:    spec-101
kind:  request (or response)
from:  educate
to:    build
ref:   — "in reply to memo #…" — the ref
The body — the payload. The mail room never opens it; only educate and build understand what's written inside.

And here is the entire mail system — one function, from lib/handoff.mjs. Eight lines.

CODE — lib/handoff.mjs 

export function writeHandoff(root, { id, kind, from, to, ref = "", title = "", body = "" }) {
  if (!id || !kind || !from || !to) throw new Error("handoff requires id, kind, from, to");
  const dir = ensureHandoffDir(root);
  const fm = [`id: ${id}`, `kind: ${kind}`, `from: ${from}`, `to: ${to}`, `ref: ${ref}`, `title: ${title}`].join("\n");
  const path = join(dir, `${id}.md`);
  writeFileSync(path, `---\n${fm}\n---\n${body}\n`);
  return path;
}
PLAIN ENGLISH

The send function. Every envelope arrives with an id, a kind, a from, a to — plus an optional ref, title, and body.

No complete address label? Refuse the envelope on the spot — undeliverable mail helps no one.

Make sure the mailbox folder (.handoff/) exists before dropping anything in.

Write the address label: six plain-text lines — id, kind, from, to, ref, title — stacked one per line.

The envelope's filename is its id: spec-101 becomes spec-101.md.

Staple the label on top (between --- fences) and drop the whole letter into the mailbox.

Hand back the path — "your mail is at…"

That's it. The entire postal service.

Notice what's not here: the mail room never reads the body. The payload is opaque — only the sender and receiver understand it.

One Copy Everywhere

Why praxis deliberately copies its shared code into all five plugins — and the robots that make copying safe

The rule we're about to break

Every programmer is taught the same commandment — DRY: never copy-paste code, share it. Praxis copies its shared code into all five plugins. On purpose. And it's the right call.

The packaging loop in the build script does three things per truck: copy the plugin, install the generator, fix the paths. Notice the dist/ folder is pure output — nobody edits it by hand.

CODE — scripts/build.mjs 

for (const plugin of targets) {

That path-fixing step matters. In development, an import says "go up two folders to find lib" — but inside the packaged truck, lib/ is one folder away. A find-and-replace, performed by code, so humans never maintain two versions of the paths.

CODE — scripts/build.mjs 

function rewriteLibImports(dir) {
  for (const e of readdirSync(dir, { withFileTypes: true })) {
    const p = join(dir, e.name);
    if (e.isDirectory()) { if (e.name !== "lib") rewriteLibImports(p); continue; }
    if (extname(p) === ".mjs") {
      const s = readFileSync(p, "utf8");
      const r = s.replaceAll("../../lib/", "../lib/");
      if (r !== s) writeFileSync(p, r);
    }
  }
}
PLAIN ENGLISH

A helper that fixes the import paths in one folder.

Only touch code files — the .mjs ones.

Read the file's text…

…replace every "up two folders to lib" with "up one"…

…and only rewrite the file if something actually changed.

…close the code-file check.

…close the loop over the folder's contents.

Done — every path now points at the truck's own generator.

The nightly binder check

Some shared content can't be imported at all. A planted template is a standalone file — at runtime it can't reach back into lib/. So the shared blocks live as literal photocopies inside the consumer files — and photocopies drift. Praxis handles it like the fleet's master recipe binder:

1
Mark it

The master and every copy carry the same "start"/"end" marker lines around each shared block.

2
Stamp it

A script — sync-shared.mjs — photocopies each marked region from the canonical source into every consumer file.

3
Compare it, every night

A test cuts the block out of the master and out of each copy and compares them, byte for byte.

4
Fail loudly

Drift fails the pre-commit hook, naming the file and the fix — before the truck ever leaves.

/** The text between the `<name>:start` and `<name>:end` marker lines (markers excluded). */
export function extractRegion(text, name) {
  const re = new RegExp(`^.*${name}:start.*$\\n([\\s\\S]*?)^.*${name}:end.*$`, "m");
  const m = text.match(re);
  return m ? m[1] : null;
}

Asking git: did the coastline move?

With a pin in every note, a dumb check can put one precise question to the ledger. The whole freshness gate boils down to a single git log question:

git log <pin>..HEAD -- <sources> "List every change to these files since that snapshot." HEAD means "now."

An empty answer means fresh. Any answer means stale — and the guilty commits get named. Here's the real check, from the real gate.

grounding-wiki/gates/freshness.mjs 

let changed = "";
try {
  changed = git(repoRoot, ["log", "--oneline", `${pin}..HEAD`, "--", ...sources]);
} catch (e) {
  fails.push(`${rel}: git log failed for its sources (${String(e).split("\n")[0]})`);
  continue;
}
if (changed) {
  const lines = changed.split("\n");
  fails.push(`${rel}: STALE — sources changed since ${pin.slice(0, 12)} (${lines.length} commit(s), e.g. ${lines[0]})`);
}
